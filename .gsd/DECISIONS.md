| D001 | M001 | arch | Binary process model | Persistent process (spawned once, stays alive, JSON over stdin/stdout) | Avoids re-parsing tree-sitter grammars per command, enables in-memory caching of parse trees, call graphs, checkpoints, and edit history | No |
| D002 | M001 | arch | LSP integration architecture | Plugin mediates LSP — binary never connects to language servers directly | Keeps binary's external surface clean (JSON in, JSON out). Plugin enriches commands with LSP data via optional `lsp_hints` fields. | No |
| D003 | M001 | arch | LSP readiness | Provider interface from M001 onward — tree-sitter default, LSP-ready | Command JSON schema has optional `lsp_hints` fields. Architecture supports LSP upgrade path without refactoring. | No |
| D004 | M001 | convention | Language priority | Web-first: TS/JS/TSX → Python → Rust → Go | Most AI agents work in TS/JS. TS/JS/TSX share ~80% of tree-sitter query patterns. Ship highest-value languages first. | Yes — if usage data shows different priority |
| D005 | M001 | arch | Safety system scope | Both per-file undo AND workspace checkpoints from M001 | Per-file undo is the immediate safety net. Workspace checkpoints are "save game" for risky multi-file changes. Both needed for agent confidence. | No |
| D006 | M001 | arch | Binary distribution timing | Local-first — build distribution pipeline as late M001 slice after tools are proven | No point building a distribution pipeline for a binary that doesn't do anything useful yet. | No |
| D007 | M001 | arch | Validation scope | Tree-sitter syntax (default, ~1ms) + opt-in full type-checker invocation (synchronous, 1-10s) | Default is fast. Full mode is synchronous because it's opt-in — agent explicitly asked for it, occasional wait is acceptable. | Yes — if full validation proves too slow in practice |
| D008 | M003 | arch | Call graph construction strategy | Lazy/incremental with file watcher for invalidation | Eager full scan too slow for large codebases. Lazy gives fast first results. File watcher keeps graph current. Worktree-aware scoping. | No |
| D009 | M001 | convention | JSON protocol format | Newline-delimited JSON (one JSON object per line) | Simple, well-understood protocol. Each request/response is self-contained. Easy to parse and debug. | No |
| D010 | M001/S01 | arch | Request parsing strategy | Two-stage: deserialize JSON envelope (id + command + flattened params), then dispatch on command string | Separates transport concerns (JSON parsing, ID tracking, error envelope) from command logic. Each command handler receives pre-validated params map. Malformed JSON is caught at stage 1 without touching command logic. | No |
| D011 | M001/S01 | convention | Integration test architecture | Persistent AftProcess struct with BufReader held over child stdout lifetime | Per-call BufReader silently loses buffered data during sequential reads. AftProcess pattern is mandatory for any test sending multiple commands to the binary. | No |
| D012 | M001/S02 | arch | Query pattern embedding | Inline const &str in parser.rs — compiled into binary | Zero external file dependencies at runtime. Simpler than include_str! for now since queries are short. Can refactor to .scm files later if queries grow. | Yes — if query patterns need separate file editing |
| D013 | M001/S02 | convention | Symbol type location | Symbol, SymbolKind, Range in `src/symbols.rs` — LanguageProvider trait stays in `src/language.rs` | Separates the data model (symbols.rs) from the trait contract (language.rs). Clean for S03/S05 which need Symbol for serialization but don't need the trait. | No |
| D014 | M001/S02 | arch | TreeSitterProvider interior mutability | RefCell<FileParser> inside TreeSitterProvider | LanguageProvider trait methods take &self (required for trait object safety), but FileParser needs &mut self for cache updates. RefCell is single-threaded safe — binary is single-threaded. | Yes — if concurrency added, switch to Mutex |
| D015 | M001/S02 | convention | tree-sitter QueryMatches iteration | streaming-iterator crate as explicit dependency | tree-sitter 0.24 QueryMatches implements StreamingIterator (not std Iterator). Must use advance()/get() pattern. Dependency is small and already in tree-sitter's dep tree. | No |
| D016 | M001/S02 | convention | Rust trait SymbolKind mapping | trait_item → SymbolKind::Interface | Traits are semantically closest to interfaces. Consistent with how LSP servers (rust-analyzer) categorize traits. Adding a separate SymbolKind::Trait would fragment downstream consumers for no gain. | Yes — if downstream consumers need trait-specific behavior |
| D017 | M001/S02 | convention | Rust impl scope chain format | Inherent: ["TypeName"], Trait: ["Trait for TypeName"] | Distinguishes inherent vs trait implementations in a single string. Agents can pattern-match "for" to detect trait impls. Parent field always points to the concrete type. | No |
| D018 | M001/S02 | arch | Python/Go scope chain strategy | Parent-node walking rather than query captures | tree-sitter queries can't express arbitrary ancestor relationships. Walking parents at extraction time is simple, correct, and fast for the shallow nesting typical in Python and Go. | No |
| D019 | M001/S02 | convention | TypeAlias as 7th SymbolKind | Added TypeAlias beyond the plan's 6 variants | TS type aliases are ubiquitous in agent workflows. Downstream consumers (outline, edit_symbol) benefit from distinguishing type aliases from other constructs. All match arms must handle 7 variants. | No |
| D020 | M001/S02 | convention | Per-language export detection strategy | TS/JS: byte-range containment in export_statement; Rust: pub modifier; Go: uppercase first char; Python: always false | Each language has different export semantics. Byte-range containment for TS/JS is simpler than parent-chain walking and handles re-exports. Python has no syntactic export marker. | No |
| D021 | M001/S03 | convention | Command module pattern | `src/commands/` directory with per-command modules (outline.rs, zoom.rs), each exporting a `handle_*` function taking `(&RawRequest, &dyn LanguageProvider) -> Response` | Separates command logic from dispatch. Each command is independently testable. Establishes the pattern S04/S05 will follow for their commands. | No |
| D022 | M001/S03 | convention | Zoom caller/callee scope | File-scoped only — calls_out and called_by limited to symbols within the same file | Cross-file call graph is M003 (R020). File-scoped analysis is fast (single AST walk) and useful for understanding local symbol relationships. Last-segment heuristic for member access calls. | Yes — M003 extends to cross-file |
| D023 | M001/S03 | arch | Zoom AST access pattern | Zoom handler creates own FileParser rather than extending LanguageProvider trait | Keeps LanguageProvider trait minimal (resolve_symbol + list_symbols). Avoids borrow conflicts with RefCell-based TreeSitterProvider. FileParser is cheap to construct — cache miss is one re-parse. | Yes — if perf requires shared cache |
| D024 | M001/S03 | convention | Call extraction algorithm | Recursive node walk with byte-range containment, language-specific node kind matching | Single code path for all 6 languages. call_expression (TS/JS/Go), call (Python), call_expression + macro_invocation (Rust). Simpler and more maintainable than per-language query patterns for call expressions. | No |
