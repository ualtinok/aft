## v0.26.1

Patch release fixing one callgraph correctness bug in monorepos where individual workspace packages have their own lockfiles.

## What was broken

`aft_navigate callers` and `aft_navigate impact` silently dropped cross-package consumers in workspaces where any package contained its own `bun.lock` or `package-lock.json` alongside the root one.

Symptom: querying callers of a function exported from `@workspace/aft-bridge` returned only some consumer packages (the ones whose tree happened to have no nested lockfile). Other consumers — making the identical import — looked invisible to the call graph.

## Root cause

`find_workspace_root` walked up the directory tree looking for either a `package.json workspaces` field OR any package lockfile (`bun.lock`, `package-lock.json`, `yarn.lock`, `pnpm-lock.yaml`). The lockfile heuristic was wrong: nested per-package lockfiles are normal in monorepos (especially with bun's per-package install mode), and they stopped the upward walk one level too early. The resolver then couldn't find the real workspace `package.json` and gave up on the import, marking the call edge as unresolved.

## What changed

Removed the lockfile heuristic. Workspace detection now relies solely on `package.json` declaring a `workspaces` field — which is what every monorepo tool actually uses to define workspace membership. Regression test added in `crates/aft/src/callgraph/caller_resolver.rs` covers the exact pattern (3-package workspace + nested `bun.lock` in one consumer).

## Upgrading

Restart your editor / OpenCode / Pi to pick up the new bridge binary. No config changes needed.

- npm: `npx --bun @cortexkit/aft setup` (or upgrade your existing install)
- Plugin: `@cortexkit/aft-opencode@0.26.1`, `@cortexkit/aft-pi@0.26.1`
- CLI: `@cortexkit/aft@0.26.1`
- Binary: `agent-file-tools@0.26.1` on crates.io and platform packages on npm
